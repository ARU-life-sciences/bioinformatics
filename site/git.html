git and GitHub

Two very important technologies for modern software development. For students and academics, these are invaluable tools to catalogue and develop your code. Many branches of academia which rely on software, like bioinformatics, use git to version control their software, and GitHub to make it available to a wide audience. This is important so other people can see what you are working on, and can lead to collaborations in longer term. In addition, many journals require source code to be published, and if they don't require it, it's a good idea to do so anyway for reproducibility. The link at the top right of this page will take you to the home of this website, which is version controlled using git and published to the world using GitHub.

As with many things, learning basic git is quite easy, but things get tough very quickly. Stick with it!

Installing git and joining GitHub

Depending on which machine you have (Windows, Mac, Linux), you can follow the instructions on the official git website to install it. Once installed, open up terminal on a Mac or Linux, or powershell on Windows, and type:

git --version

On Windows machines, I'd recommend installing WSL (Windows Subsystem for Linux), which should have git installed on it already. Failing that, git BASH is a good alternative. Install that here. Provided that you are not crippled by the security system of your computer, this should be relatively easy to do.

Once you have installed git, we can look at GitHub. You can sign up for a free account on the GitHub website. It should be very self-explanatory to sign up. If you are part of ARU, please join the Life Sciences GitHub.

Explore a repository

Once you have signed up for GitHub, you can explore some repositories. I recommend looking at the Bioinformatics repository on the ARU-life-sciences GitHub. You can see all the files and folders.

Creating a new repository

There are two ways to create a new repository. You can create a new repository on GitHub and clone it to your local machine, or you can create a new repository on your local machine and push it to GitHub. I recommend the latter, as I think it's easier to do.

You firstly need to create a directory/folder on your machine. In terminal or powershell, navigate to the directory and type: 

git init

This will create a new git repository in your current directory. This is a local repository. Now you can add, modify, or delete files in the directory and they can be tracked. These files are not tracked by default - we have to do the work.

Now go over to GitHub and create a new repository. It will ask for some details, so fill those in. Once you click 'create repository', there will be some commands you can run to link your repository you made on your computer just now, to the remote one.

The code will be something like the following:

git remote add origin https://github.com/USERNAME/REPO NAME.git
git branch -M main
git push -u origin main

git remote add origin tells your computer where the remote repository is (i.e. on GitHub). After running this, the lcoal repository is linked to the remote one. git branch -M main renames the default branch from master to main. We will consider branches much later, but for now think of them as copies of the repository under different names. git push -u origin main pushes the local repository to the remote one. If you run this command, it will fail as there's nothing in our repository! But don't worry, we will fix this in a moment.

Basic idea

We have made a blank repository, and we want to put something in it. We can create a file in the directory. Do this any way you want. Here's a way I'd do it on a Unix like machine:

echo "Hello, world!" > hello.txt

Now we have a simple text file with the text "Hello, world!" in it. We can add this file to the "local stage" using:

git add hello.txt

This adds the file to the local stage, which is like a waiting area. We can add more files to the stage if we want to. A core idea in git is that we make incremental changes to our repository which we can easily reason with. Adding a file to the stage is necessary, but doesn't actually do anything by itself. We need to commit these changes. Commiting changes adds the change to our local log file. Let's do it:

git commit -m "Add hello.txt"

The -m flag is for "message". The message is a short description of the change we made, and is necessary. Please make your commit messages informative and descriptive if you can.

We have added and commited our changes to the local repository. Check the message log using git log. You should see the message you just made. 

git log

We can now push these changes to the remote repository (i.e. GitHub) using:

git push -u origin main

The changes are now on GitHub! You can check by refreshing the page.

Adding a README

It's good practice to add a README file to your repository. This file can contain information about the repository, such as what it's for, who made it, and how to use it. This file will be displayed on the repository page on GitHub. We will switch gear and add it from the website.

If you navigate to your repository on the GitHub website, you can click the "Add a README" button. This will bring up a text editor which you can use to write your README. Put something in there and save it. Notice how you will have to commit the changes just like we did before. You can add an extra description if you wish. After you are done, commit the changes.

Now we have our changes committed to the remote (on GitHub), but they are not on our local machine. We can pull them in using:

git pull

The changes should now be merged with your local repository. You can check using:

git log

It should show that you now have two commits.

And that's it! You are now a git and GitHub user. This was a very simple introduction to version control. We will look at multi-user collaboration, branches, and pull requests in a separate tutorial. Good luck!
